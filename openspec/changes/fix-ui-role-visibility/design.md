## Context

本变更包含两类跨模块问题：

- “你的行动”页面布局：当前“你的行动”长文本会被下方“游戏记录”遮挡，说明页面缺少稳定的可滚动容器与可控的区域分配。
- 信息可见性：当前客户端/服务端对身份牌、心声、夜晚狼人私聊等信息缺少按“阶段 + 阵营 + 目标玩家”的权限约束，导致玩家在对局阶段获取不应获得的信息（包括对立阵营私密内容），破坏公平性与推理玩法。

该变更的目标是在不改变核心玩法流程的前提下，修复 UI 可用性与信息泄露问题，并补齐狼人阵营的队友识别与队内信息共享规则。

## Goals / Non-Goals

**Goals:**

- “你的行动/游戏记录”采用上下分割布局，并支持鼠标拖拽调整高度；两面板都具备独立滚动，长内容不会遮挡。
- 对局阶段信息可见性满足：
  - 玩家从回合开始即可看到自己的本轮身份牌。
  - 非复盘阶段：默认只能看到他人的公开发言；不可见他人的身份牌与心声；对立阵营私密沟通不可见。
  - 复盘阶段：心声等复盘信息可见。
  - 当玩家为狼人阵营：可识别狼人队友（头像特殊标记），并可看到狼人队友的心声与发言，且可见狼人夜晚聊天。
- 从源头降低信息泄露风险（优先服务端裁剪，其次客户端严格过滤与渲染约束）。

**Non-Goals:**

- 不调整既有回合规则、胜负条件、AI 决策策略与角色配置。
- 不在本次变更中引入全新的房间/对局状态机；仅在现有状态中补齐“复盘阶段”标识或沿用已有标识。
- 不实现复杂的布局持久化/多端同步（如需仅做本地记忆，且不作为强依赖）。

## Decisions

1) “你的行动/游戏记录”采用可拖拽分割面板（垂直方向）

- 方案：使用单一父容器（占满可用高度），内部为两块面板 + 分割条（drag handle）。面板使用 flex column 布局，分割条通过 pointer/mouse 事件驱动调整上/下面板的高度（像素值或比例值）。
- 关键约束：
  - 上下两面板都有最小高度（防止被拖到不可用）。
  - 两面板内容各自 overflow: auto，避免“遮挡”而不是“滚动”。
  - 拖拽时禁用文本选择，并在 pointerup 释放监听，避免泄漏事件监听与卡顿。
- 可选增强：将面板比例写入 localStorage（或项目现有持久化机制），下次打开保持用户偏好；如果无统一持久化方案则先不做。

2) 信息可见性以“服务端裁剪优先、客户端兜底过滤”为原则

- 服务端裁剪（优先）：根据请求方玩家与对局阶段，仅返回其“应该看到”的字段：
  - 身份牌：仅返回自己的身份；对局阶段不返回他人身份（复盘除外）。
  - 心声：对局阶段仅在“狼人视角 && 目标为狼人队友”时返回；复盘阶段返回全部。
  - 狼人夜聊：仅“狼人视角”返回，且仅在夜晚/相关阶段返回；非狼人视角不返回。
- 客户端过滤（兜底）：即使服务端暂时仍下发全量数据，UI 也必须按同一套规则决定“是否渲染/是否可展开/是否可进入”对应内容，避免通过界面直接读到敏感信息。
- 理由：仅靠客户端隐藏无法防止调试工具/抓包泄露；仅靠服务端裁剪在迭代期可能出现接口不一致，因此采用双层防线。

3) 自身份牌“回合开始即获得”，并从 UI/状态层避免“未知”展示

- 方案：以“当前玩家的本轮角色”作为对局数据的必填项（从回合开始就存在），UI 显示层对自己永远走“已知”分支。
- 若存在“身份揭示动画/延迟”：允许展示过渡态（例如 loading/翻牌动画），但最终态不应为“未知”。

4) 狼人队友头像特殊标记以“可识别但不影响布局”为原则

- 方案：在玩家列表项上增加一个仅对狼人可见的标识（如角标/描边/徽章）。该标识只依赖“当前玩家阵营 + 目标玩家是否为狼人队友”两个条件。
- 同时将“可见的心声/夜聊入口”与该条件保持一致，避免标识存在但信息仍不可见、或相反导致体验割裂。

## Risks / Trade-offs

- [仅客户端隐藏导致信息泄露] → 优先服务端裁剪敏感字段；客户端作为兜底仍实现严格渲染门禁。
- [拖拽面板与滚动冲突] → 分割条单独命中区域（足够高度/宽度），拖拽时阻止默认行为并临时禁用选择；面板内容滚动区域不拦截拖拽事件。
- [最小高度设置不当导致内容不可用] → 设置合理的 minHeight（例如基于容器高度的百分比下限或固定像素），并在窗口 resize 时重新 clamp。
- [复盘阶段定义不清] → 统一使用“对局阶段枚举”或已有字段；缺失时以“对局已结束 && 进入复盘视图”为复盘条件作为兼容。
- [服务端改动影响旧客户端] → 接口字段保持向后兼容：敏感字段可缺省/为空，客户端以缺省处理，不要求字段必有。

## Migration Plan

- 第一步：客户端实现分割面板与渲染门禁（不依赖服务端变更也能阻止 UI 泄露）。
- 第二步：服务端增加按视角裁剪逻辑（或新增“viewerProjection”类响应），逐步灰度；客户端继续兼容“全量/裁剪”两种响应。
- 第三步：上线后补充监控与回归用例（关键场景：好人视角看不到狼人夜聊；狼人视角能看到队友标识与队友心声；复盘阶段能看到心声）。
- 回滚：若服务端裁剪出现问题，可先关闭裁剪开关回退到全量下发，但客户端门禁保留以维持公平性。

## Open Questions

- 当前项目对“复盘阶段”的状态字段与进入时机是什么（是否已有 gamePhase / isReplay / isReview 标识）？
- “狼人夜聊”在现有实现中属于独立频道还是复用聊天记录结构（需要确定裁剪与 UI 入口位置）？
- 身份牌数据的来源：是开局一次性分配，还是每回合变化（“本轮身份牌”与“整局身份”是否一致）？
